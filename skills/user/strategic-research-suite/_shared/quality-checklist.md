# Quality Assurance Checklist - Strategic Research Suite

**研究品質を保証するための統合チェックリスト**

このチェックリストは、Strategic Management Journal、Academy of Management Journal、Organization Science等のトップジャーナル基準に基づいています。

---

## 📋 Phase別チェックリスト

### Phase 1: Research Design ✓

#### 理論的基盤
- [ ] **RQ明確性**: Research Questionが具体的で検証可能
- [ ] **理論的貢献**: 既存理論への新規性が明確
- [ ] **仮説導出**: 理論から論理的に仮説を導出
- [ ] **境界条件**: 理論が適用される条件を明示

#### 変数定義
- [ ] **操作的定義**: すべての変数の測定方法が明確
- [ ] **構成概念妥当性**: 変数が理論的概念を適切に捉えている
- [ ] **先行研究との整合性**: 標準的な測定方法を使用

---

### Phase 2: Data Source Discovery ✓

#### データ適合性
- [ ] **カバレッジ**: 研究対象（国・業界・期間）を十分にカバー
- [ ] **変数完備性**: 必要な変数がすべて取得可能
- [ ] **時間範囲**: 最低5年以上の観測期間
- [ ] **サンプルサイズ**: 統計的検出力に十分（N×T ≥ 1000推奨）

#### データ信頼性
- [ ] **出典信頼性**: 査読論文で使用実績あり
- [ ] **更新頻度**: データが定期的に更新されている
- [ ] **アクセス可能性**: 再現のためにアクセス方法が明確

---

### Phase 3: Data Collection Strategy ✓

#### 収集計画
- [ ] **優先順位**: データソースの優先順位が明確
- [ ] **バックアップ**: 代替データソースを確保
- [ ] **スケジュール**: 現実的な収集スケジュール
- [ ] **リスク管理**: データアクセス失敗時の対応策

#### 技術的準備
- [ ] **API認証**: 必要な認証情報を取得済み
- [ ] **レート制限**: API制限を確認・対応策あり
- [ ] **ストレージ**: 十分なストレージ容量確保

---

### Phase 4: Panel Dataset Construction ✓

#### データ構造
- [ ] **パネル構造**: (firm_id, year) のMultiIndexが正しい
- [ ] **一意性**: 各(firm_id, year)が一意
- [ ] **時間の連続性**: 欠損年がないか確認（または意図的）

#### データ品質
- [ ] **欠損値処理**: 欠損値の原因を特定・対処方針決定
- [ ] **異常値検出**: Box plot, Z-score等で検出
- [ ] **整合性**: 関連変数間の整合性確認（例: ROA = NI / Assets）

---

### Phase 5: Quality Assurance ✓

#### 統計的品質
- [ ] **Benford's Law**: 財務数値が期待分布に従うか
- [ ] **構造変化**: Chow test等で構造変化を検定
- [ ] **多重共線性**: VIF < 10 を確認
- [ ] **異分散性**: White test, Breusch-Pagan test

#### サンプル代表性
- [ ] **選択バイアス**: サンプル選択の非ランダム性を確認
- [ ] **生存バイアス**: 廃業企業の除外がバイアスを生んでいないか
- [ ] **業界分布**: 実際の産業構造を反映しているか

#### 統計的検出力
- [ ] **検出力分析**: Power ≥ 0.80 を確認
- [ ] **効果量推定**: 先行研究から期待効果量を推定
- [ ] **サンプルサイズ**: 必要サンプルサイズを満たしているか

---

### Phase 6: Variable Construction ✓

#### 変数の妥当性
- [ ] **理論的妥当性**: 変数が理論的概念を適切に測定
- [ ] **先行研究整合性**: 標準的な測定方法を使用
- [ ] **スケール**: 変数のスケール（%、ratio等）が適切

#### 変数変換
- [ ] **対数変換**: 歪度の大きい変数を対数変換
- [ ] **標準化**: 必要に応じて標準化（特に交互作用項）
- [ ] **ラグ**: 因果の時間順序を考慮してラグ設定

#### 統制変数
- [ ] **網羅性**: 重要な交絡要因を統制
- [ ] **過剰統制回避**: Collider, Mediatorを統制変数に含めない
- [ ] **業界・年効果**: 産業ダミー・年ダミーの必要性確認

---

### Phase 7: Statistical Analysis ✓

#### モデル選択
- [ ] **Hausman test**: FE vs. RE の選択を検定で判断
- [ ] **モデル適合度**: R², AIC, BICで比較
- [ ] **残差診断**: 正規性、等分散性を確認

#### 内生性対策
- [ ] **内生性の特定**: 逆因果、欠落変数を検討
- [ ] **対策手法**: IV, PSM, Heckman等を適切に選択
- [ ] **操作変数の妥当性**: IV relevance & exogeneity確認

#### 頑健性チェック
- [ ] **代替変数**: DVやIVの代替測定を試行
- [ ] **サブサンプル**: 業界別、規模別で再分析
- [ ] **異常値除外**: Winsorization前後で結果確認
- [ ] **ラグ構造**: 異なるラグ設定で再分析

---

### Phase 8: Documentation & Reproducibility ✓

#### データ系譜
- [ ] **収集手順**: データ収集の全手順を文書化
- [ ] **変数定義**: すべての変数の計算式を記載
- [ ] **除外基準**: サンプル除外の理由を明記

#### コード品質
- [ ] **コメント**: コードに十分なコメント
- [ ] **モジュール化**: 機能ごとに関数化
- [ ] **エラーハンドリング**: 例外処理を実装

#### 再現可能性
- [ ] **Docker環境**: 環境を再現可能に
- [ ] **requirements.txt**: パッケージバージョン固定
- [ ] **実行手順**: README.mdに実行手順明記
- [ ] **テスト**: pytest等で主要処理をテスト

## 📊 トップジャーナル別追加基準

### SMJ (Strategic Management Journal)

**データ要件**:
- [ ] サンプルサイズ: N×T ≥ 1000
- [ ] 観測期間: ≥ 5年
- [ ] 複数データソース: 財務+非財務データ統合

**分析要件**:
- [ ] 内生性対策: 必須（IV, PSM, FE等）
- [ ] Robustness checks: 3つ以上
- [ ] メカニズム検証: 媒介分析推奨

**執筆要件**:
- [ ] 理論的貢献: 明確に記述
- [ ] 実務的示唆: 含める
- [ ] 限界の議論: 誠実に記述

---

### AMJ (Academy of Management Journal)

**理論要件**:
- [ ] 理論的新規性: 強く要求
- [ ] 仮説の論理性: 詳細な理論展開
- [ ] 境界条件: 明示

**方法論要件**:
- [ ] 統計的検出力: Power ≥ 0.80
- [ ] 効果量報告: 必須（β, R²増分）
- [ ] 代替説明: 検討必須

**執筆要件**:
- [ ] 理論セクション: 充実した記述
- [ ] 貢献の明確化: Theoryへの貢献明示
- [ ] 将来研究: 具体的な提案

---

### OS (Organization Science)

**データ要件**:
- [ ] データの新規性: 重視
- [ ] 詳細データ: 企業レベルの詳細情報

**分析要件**:
- [ ] メカニズム検証: 媒介分析推奨
- [ ] 異質性分析: サブグループ分析推奨
- [ ] ダイナミクス: 時間的変化の分析

**執筆要件**:
- [ ] 組織論的視点: 組織プロセスの考察
- [ ] 理論的精緻化: 詳細なメカニズム議論

## 🔍 Common Pitfalls（よくある落とし穴）

### データ収集段階

❌ **落とし穴1**: 生存バイアスを無視
```
問題: 廃業企業を除外 → 成功企業のみ分析
対策: 廃業企業データも可能な限り収集
```

❌ **落とし穴2**: 選択バイアスを無視
```
問題: データ利用可能な企業のみ分析
対策: Selection model（Heckman）で対処
```

---

### 変数構築段階

❌ **落とし穴3**: 時間の因果順序を無視
```
問題: 同時点のX→Y関係を分析
対策: X_t-1 → Y_t のようにラグ設定
```

❌ **落とし穴4**: 過剰統制
```
問題: 媒介変数を統制変数に含める
対策: 媒介変数は別途媒介分析で検証
```

---

### 統計分析段階

❌ **落とし穴5**: 内生性を無視
```
問題: 単純なOLS回帰のみ実施
対策: 内生性の可能性を議論し、対策を講じる
```

❌ **落とし穴6**: 多重検定問題
```
問題: 多数の仮説を検定 → 偶然の有意性
対策: Bonferroni補正、FDR調整
```

---

### 執筆段階

❌ **落とし穴7**: p-hackingの印象
```
問題: 有意な結果のみ報告
対策: 事前登録、全結果報告（Appendix活用）
```

❌ **落とし穴8**: 因果表現の不適切使用
```
問題: 相関を因果と表現
対策: 「関連」「予測」等の表現を使用（因果デザインでない限り）
```

## 🎯 品質スコアリング

各Phaseの達成度を数値化：

```python
def calculate_quality_score(checklist_results):
    """
    品質スコア = Σ(達成項目数 / 総項目数) / Phase数
    
    80%以上: 投稿可能レベル
    60-80%: 追加作業必要
    60%未満: 大幅な見直し必要
    """
    total_score = sum([achieved / total for phase in checklist_results])
    return total_score / len(checklist_results)
```
