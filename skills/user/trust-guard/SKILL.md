# TrustGuard: 情報信頼性評価・ハルシネーション防止システム

## スキル概要

**名称**: TrustGuard (Information Reliability Assessment & Hallucination Prevention System)  
**バージョン**: 1.0.0  
**最終更新**: 2025-11-22  
**カテゴリ**: 学術研究支援、品質保証、事実検証

### 目的

学術研究および情報分析において、情報源の信頼性を多層的に検証し、ハルシネーション(事実に基づかない情報生成)を自動的に防止することで、研究の学術的厳密性と再現可能性を担保します。

### 主要機能

1. **多層検証フレームワーク** - 4段階の独立した検証プロセス
2. **信頼性スコアリング** - 客観的指標に基づく0-100点評価
3. **ハルシネーション検出** - AI生成コンテンツの自動検証
4. **代替情報源探索** - 信頼性の高い代替ソースの提案

### 想定削減時間

- 手動ファクトチェック: 1時間 → 15分
- 信頼性評価: 30分 → 5分
- 代替ソース探索: 1時間 → 20分
- **週間削減時間: 約5-7時間**

## アクティベーショントリガー

このスキルは以下の状況で自動的に起動します:

### 必須起動(自動実行)
- ユーザーが情報源の検証を明示的に要求した場合
- 学術論文、研究報告書の作成時
- 引用や参考文献の追加時
- 事実的主張に対する検証が必要な場合

### 推奨起動(提案後実行)
- 統計データや数値情報の提示時
- 専門的知識や技術情報の説明時
- 歴史的事実や時事情報の言及時
- 複数の情報源からの情報統合時

## コア機能詳細

### 1. 多層検証フレームワーク

#### 1.1 存在検証層(Existence Verification)

**目的**: 情報源の実在性と接続可能性を確認

**実行手順**:
```
1. URL構文検証
   - プロトコル(https://, http://)の確認
   - ドメイン名の妥当性チェック
   - パス構造の正確性確認

2. 複数検索エンジンでの確認
   - Google検索での存在確認
   - Bing検索での交差検証
   - 学術データベース(該当する場合)での検索

3. 接続テスト
   - web_fetch ツールでの実際の接続試行
   - HTTPステータスコード確認(200 = OK)
   - リダイレクト追跡と最終URLの記録

4. アーカイブ確認
   - Internet Archive (Wayback Machine) での過去バージョン確認
   - 削除された情報源の復元可能性評価
```

**出力例**:
```
✅ 存在検証: 合格
   - URL接続: 成功(HTTP 200)
   - 最終確認日: 2025-11-22
   - アーカイブ: 利用可能(2024年から5バージョン)
```

#### 1.2 内容検証層(Content Verification)

**目的**: 引用内容と原文の完全一致を確認

**実行手順**:
```
1. 原文取得
   - web_fetch で完全なページコンテンツ取得
   - テキスト抽出と正規化(空白、改行の標準化)

2. 引用箇所の特定
   - 完全一致検索(引用符内の正確なテキスト)
   - 近似一致検索(軽微な表記揺れの許容)
   - 文脈の前後50文字抽出

3. 一致度評価
   - 完全一致: 100%
   - 高一致(95%以上): 軽微な句読点の違い
   - 中一致(85-95%): 表記揺れ
   - 低一致(85%未満): 要再確認

4. 文脈整合性チェック
   - 引用が文脈から切り離されていないか
   - 意味の歪曲や誤解を招く引用でないか
```

**出力例**:
```
✅ 内容検証: 合格(一致度: 98%)
   - 原文箇所: 第3段落、2-4行目
   - 差異: 句読点のみ(「、」→「，」)
   - 文脈整合性: 問題なし
```

#### 1.3 整合性検証層(Consistency Verification)

**目的**: 他の確立された事実との矛盾を検出

**実行手順**:
```
1. 既知事実データベースとの照合
   - 歴史的事実(年号、人物、出来事)
   - 科学的事実(定数、法則、定理)
   - 統計データ(公式統計との照合)

2. 複数情報源の交差検証
   - 同一主張の独立した3つ以上の情報源確認
   - 矛盾する情報源の特定と評価
   - コンセンサスレベルの判定

3. 論理的整合性チェック
   - 因果関係の妥当性
   - 時系列の整合性
   - 数値の論理的妥当性(桁数、単位、比率)

4. ドメイン専門知識との照合
   - 専門用語の正確な使用
   - 理論的枠組みとの整合性
   - 業界標準や慣習との一致
```

**出力例**:
```
✅ 整合性検証: 合格
   - 交差検証: 5つの独立情報源で一致
   - 論理的整合性: 問題なし
   - 専門知識適合: 標準的な用語使用を確認
   
⚠️ 注意事項: 
   - 数値が2020年時点のものである点に留意
   - 最新データ(2024)では15%増加の可能性
```

#### 1.4 文脈検証層(Context Verification)

**目的**: 情報が元の文脈から不適切に分離されていないか確認

**実行手順**:
```
1. 文脈範囲の特定
   - 引用前後の段落(最低3段落)取得
   - セクション全体の主題確認
   - 著者の主張の全体的方向性把握

2. 選択的引用の検出
   - 反対意見や留保条件の省略チェック
   - 限定条件や前提の確認
   - 「しかし」「ただし」等の接続詞の有無

3. 意図的歪曲の検出
   - 著者の主張と引用の整合性
   - 皮肉や反語表現の見落としチェック
   - 仮定的記述の断定的引用の検出

4. 文脈依存性の評価
   - 引用単独での理解可能性
   - 追加説明の必要性判定
   - 文脈情報の補足提案
```

**出力例**:
```
✅ 文脈検証: 合格
   - 文脈整合性: 著者の主張と一致
   - 限定条件: なし(一般的主張として引用可能)
   
📝 推奨補足:
   - 引用の前に「○○の文脈において」を追加推奨
   - 著者は△△を前提としている点を明示すべき
```

### 2. 信頼性スコアリングシステム

#### 2.1 情報源タイプ別基礎スコア

```
【Tier 1: 最高信頼性】(基礎スコア: 85-100点)
- 査読済み学術論文(トップジャーナル): 95-100点
- 政府公式統計・白書: 90-95点
- 国際機関公式報告(UN, World Bank, IMF等): 90-95点
- 確立された学術書(大学出版会等): 85-90点

【Tier 2: 高信頼性】(基礎スコア: 70-84点)
- 査読済み学術論文(一般ジャーナル): 80-84点
- 主要シンクタンク報告書: 75-80点
- 大手メディアの調査報道: 75-80点
- 専門家による書籍: 70-75点

【Tier 3: 中程度信頼性】(基礎スコア: 50-69点)
- 業界団体レポート: 65-69点
- 企業公式発表(IR資料等): 60-65点
- 一般メディアのニュース記事: 55-60点
- 専門家ブログ: 50-55点

【Tier 4: 要注意】(基礎スコア: 30-49点)
- 一般ブログ・個人サイト: 40-49点
- SNS投稿: 35-40点
- 匿名掲示板: 30-35点

【Tier 5: 不適格】(基礎スコア: 0-29点)
- 検証不能な情報源: 0-29点
```

#### 2.2 修正要素(±20点)

**加点要素** (+1～+20点):
```
+10点: DOI等の永続的識別子あり
+8点: 複数の独立情報源で確認可能
+5点: 最近5年以内の発行
+5点: 著者の専門性が明確(博士号、専門職等)
+3点: 引用数が多い(学術論文の場合)
+3点: 一次情報源である
+2点: 詳細なデータソース明記
+2点: 方法論が明示的
```

**減点要素** (-1～-20点):
```
-20点: 情報源の存在を確認できない
-15点: 明確な事実誤認を含む
-10点: 重大な論理的矛盾あり
-8点: 利益相反の可能性(未開示)
-5点: 発行から10年以上経過(分野による)
-5点: 著者の専門性が不明
-3点: 二次・三次情報源
-2点: 感情的・煽動的表現多用
```

#### 2.3 総合信頼性スコア算出

```
総合スコア = 基礎スコア + 加点要素 - 減点要素

【スコア解釈】
90-100点: 極めて信頼性が高い(そのまま引用可能)
75-89点: 高い信頼性(一般的な学術利用に適格)
60-74点: 中程度の信頼性(補足的情報源として利用可)
40-59点: 低い信頼性(要検証、代替ソース推奨)
0-39点: 不適格(引用すべきでない)
```

**出力例**:
```
📊 信頼性スコア: 87点(高い信頼性)

【内訳】
基礎スコア: 80点(査読済み学術論文)
加点: +10点(DOIあり)
      +5点(2023年発行)
      +3点(引用数125件)
減点: -8点(利益相反の可能性あり※)
      -3点(二次データ使用)

※著者の所属企業が研究対象業界に関連

【評価】
✅ 一般的な学術利用に適格
⚠️ 利益相反に留意した引用を推奨
```

### 3. ハルシネーション検出システム

#### 3.1 固有名詞検証

**目的**: 実在しない人名、組織名、地名等の検出

**検証プロセス**:
```
1. 固有名詞の抽出
   - 大文字で始まる単語列
   - 人名パターン(姓名、肩書)
   - 組織名パターン(株式会社、Inc.等)
   - 地名パターン(市、県、州等)

2. 実在性確認
   - Wikipedia検索
   - Google検索(完全一致)
   - 専門データベース検索(人物、企業、地理)

3. 文脈妥当性チェック
   - 時代的整合性(歴史的人物の生没年等)
   - 地理的整合性(実在する地名の組み合わせ)
   - 組織的整合性(実在する部署、役職)

4. 結果分類
   - 確認済み: 複数の独立情報源で確認
   - 疑義あり: 1つの情報源のみ、または矛盾あり
   - 確認不能: いずれの情報源でも確認できず
```

**出力例**:
```
🔍 固有名詞検証結果:

✅ 確認済み(3件)
   - 田中太郎(東京大学教授): Wikipedia, 大学公式サイトで確認
   - 株式会社ABC: 法人番号公表サイトで確認
   - 渋谷区: 公式地名

❌ 確認不能(1件)
   - 「XYZプロジェクト」: いずれの情報源でも確認できず
   → 架空のプロジェクト名の可能性あり
   → 代替表現: 「ある大規模プロジェクト」等を推奨
```

#### 3.2 数値・統計検証

**目的**: 不正確または架空の数値データの検出

**検証プロセス**:
```
1. 数値の抽出と分類
   - 統計値(平均、中央値、割合等)
   - 絶対数(人口、売上高等)
   - 成長率・変化率
   - ランキング・順位

2. 妥当性チェック
   - 桁数の妥当性(常識的範囲内か)
   - 単位の正確性(千、百万、億等)
   - 比率の論理性(合計100%、0-100%範囲内)
   - 時系列の整合性(前年比等)

3. 公式統計との照合
   - 政府統計データベース検索
   - 国際機関統計との照合
   - 業界団体公式データとの比較

4. 矛盾検出
   - 同一主題の異なる数値
   - 計算上の矛盾(合計と内訳の不一致等)
   - 常識的範囲外の数値
```

**出力例**:
```
📊 数値検証結果:

✅ 検証済み(2件)
   - 「日本の人口1.26億人」: 総務省統計局で確認(2023年)
   - 「GDP成長率2.1%」: 内閣府公式データと一致

⚠️ 要確認(1件)
   - 「業界シェア45%」: 
     → 公式統計では確認できず
     → 出典が業界推計値である点を明記推奨
     → 「推計」または「約」の表現追加を推奨

❌ 矛盾検出(1件)
   - 「前年比120%増加で売上3000万円」
     → 前年売上1364万円の計算だが、同じ文書内で
        「前年売上2000万円」との記述あり
     → 数値の再確認が必要
```

#### 3.3 引用存在性検証

**目的**: 実在しない情報源への引用の検出

**検証プロセス**:
```
1. 引用形式の確認
   - 著者名の存在
   - タイトルの存在
   - 発行年の存在
   - URL(オンライン資料の場合)

2. 情報源の検索
   - 学術データベース(Google Scholar, PubMed等)
   - 書籍データベース(Amazon, 国会図書館等)
   - ニュースアーカイブ
   - 組織公式サイト

3. 一致度評価
   - 完全一致: タイトル、著者、年がすべて一致
   - 部分一致: 一部の要素のみ一致
   - 不一致: いずれの要素も確認できず

4. 代替ソース提案
   - 類似の主張をする別の情報源
   - より信頼性の高い代替情報源
   - 一次情報源への誘導
```

**出力例**:
```
📚 引用検証結果:

✅ 確認済み(2件)
   - Smith, J. (2022) "Digital Transformation": 
     Google Scholarで確認、被引用数45件

❌ 確認不能(1件)
   - 「山田花子(2021)『AI時代のマーケティング』」:
     → 国会図書館、Amazon、Google Scholarいずれでも確認できず
     → 著者名・タイトルの誤記の可能性
     
   💡 代替ソース提案:
   - 類似主題の信頼できる文献:
     「佐藤太郎(2021)『デジタルマーケティング入門』東洋経済新報社」
     (信頼性スコア: 78点)
```

#### 3.4 確証バイアス検出

**目的**: 一方的な情報選択や偏った解釈の検出

**検証プロセス**:
```
1. 視点の多様性評価
   - 支持する情報源と反対する情報源の比率
   - 異なる立場からの視点の有無
   - 限定条件や留保の明示度

2. 反証の探索
   - 主張に対する反対意見の検索
   - 矛盾する研究結果の確認
   - 代替説明の可能性検討

3. 情報源の偏り検出
   - 特定の組織・著者への過度の依存
   - 同一視点の情報源の重複
   - 地理的・文化的偏り

4. バランス提案
   - 対立する視点の提示
   - 論争的論点の明示
   - 不確実性の明確化
```

**出力例**:
```
⚖️ 確証バイアス検証:

⚠️ 偏りの可能性あり

【検出内容】
- 5つの情報源すべてが同一の結論を支持
- 反対意見や留保条件の記述なし
- 情報源3件が同一組織からの発表

💡 改善提案:
1. 反対意見の追加:
   - Johnson et al. (2023)は異なる結論を提示
   - 「一方で、〇〇という批判もある」の追加推奨

2. 限定条件の明示:
   - 「特定の条件下では」等の表現追加
   - 「さらなる研究が必要」との留保推奨

3. 情報源の多様化:
   - 独立した研究機関の報告書を追加
   - 異なる地域・文化圏の研究を参照
```

### 4. 代替情報源自動探索

#### 4.1 代替ソース探索プロトコル

**起動条件**:
```
- 信頼性スコア60点未満の情報源
- URL接続不能の情報源
- 検証で矛盾が検出された情報源
- ユーザーの明示的要求
```

**探索手順**:
```
1. 主題とキーワードの特定
   - 中心的概念の抽出
   - 関連用語の生成
   - 検索クエリの最適化

2. 多段階検索
   【第1段階: 学術情報源】
   - Google Scholar検索
   - PubMed検索(医学・生命科学)
   - SSRN検索(社会科学)
   
   【第2段階: 公式情報源】
   - 政府公式サイト検索
   - 国際機関データベース
   - 業界団体公式サイト
   
   【第3段階: 高品質メディア】
   - 主要新聞社データベース
   - 専門誌アーカイブ
   - シンクタンク報告書

3. 候補の評価
   - 信頼性スコア算出
   - 内容の関連性評価
   - アクセス可能性確認

4. 最適ソースの推奨
   - 上位3-5件の提示
   - 各ソースの強みと限界の説明
   - 使用場面の推奨
```

**出力例**:
```
🔄 代替情報源の探索結果:

【元の情報源の問題点】
- 信頼性スコア: 42点(低い)
- URL接続不能(404 Not Found)
- 著者の専門性不明
```

## 統合ワークフロー

### 標準的な検証プロセス(4段階)

```
【Stage 1: 初期スキャン】(所要時間: 1-2分)
└─ 情報源タイプの識別
└─ 基礎スコアの算出
└─ 明らかな問題の検出(URL不通、形式エラー等)

【Stage 2: 多層検証】(所要時間: 3-5分)
└─ 存在検証: URL接続テスト、検索エンジン確認
└─ 内容検証: 引用と原文の照合
└─ 整合性検証: 交差検証、論理チェック
└─ 文脈検証: 文脈の適切性確認

【Stage 3: 詳細分析】(所要時間: 5-10分)
└─ 信頼性スコア最終算出
└─ ハルシネーション検出(固有名詞、数値、引用)
└─ 確証バイアス検出
└─ 代替ソース探索(必要な場合)

【Stage 4: レポート生成】(所要時間: 2-3分)
└─ 検証結果の統合
└─ 推奨アクションの提示
└─ 代替情報源リストの作成
└─ 最終評価とサマリー
```

### 緊急モード(クイック検証)

重要度の低い情報や時間制約がある場合:

```
【所要時間: 2-3分】
1. URL接続確認のみ
2. 情報源タイプの識別
3. 基礎スコアの算出
4. 明らかな問題の指摘

※詳細検証は後で実施可能
```

## ベストプラクティス

### Do's (推奨事項)

✅ **検証は早い段階で実施**
   - 執筆開始前に主要情報源を検証
   - 引用追加の都度、即座に検証

✅ **複数の独立情報源を使用**
   - 同一主張を3つ以上の情報源で確認
   - 異なるタイプの情報源を組み合わせ

✅ **一次情報源を優先**
   - 可能な限り原著論文や公式統計を直接引用
   - 二次情報源は補足的使用に限定

✅ **限定条件を明示**
   - 研究の前提条件や限界を明記
   - 適用範囲を明確化

✅ **対立する見解も提示**
   - 論争的論点では複数の立場を紹介
   - balanced な記述を心がけ

### Don'ts (禁止事項)

❌ **未検証情報の使用**
   - TrustGuardを通さない引用は禁止
   - 「おそらく正しいだろう」での引用は厳禁

❌ **単一情報源への依存**
   - 重要な主張は必ず複数ソースで確認
   - 特に数値データは交差検証必須

❌ **文脈を無視した引用**
   - 元の文脈から切り離した引用は避ける
   - 限定条件や留保を省略しない

❌ **信頼性スコア60点未満の単独使用**
   - より信頼性の高い代替ソースを探索
   - やむを得ず使用する場合は限界を明記

❌ **確証バイアスの放置**
   - 自分の主張を支持する情報のみ集めない
   - 反対意見や矛盾する証拠も積極的に探す

## トラブルシューティング

### よくある問題と解決策

#### 問題1: URL接続不能(404 Not Found)

**原因**:
- ページの削除や移転
- URLの誤記
- 一時的なサーバーエラー

**解決策**:
```
1. 再度接続試行(一時的エラーの可能性)
2. URLの綴りを確認(特に大文字小文字、記号)
3. Internet Archive (Wayback Machine) で過去バージョン確認
4. Google検索で同一コンテンツの代替URL探索
5. DOIがある場合、DOI経由でのアクセス試行
6. 著者・組織の公式サイトで再公開版を確認
```

#### 問題2: 引用内容が原文で見つからない

**原因**:
- 引用の誤記
- 翻訳による表現の変化
- 二次情報源からの不正確な引用

**解決策**:
```
1. キーワードを抜き出して部分一致検索
2. 類似表現で検索(同義語、言い換え)
3. 原文が他言語の場合、翻訳版も確認
4. ページ番号が指定されている場合、その前後も確認
5. 一次情報源に直接あたる(二次引用を避ける)
```

#### 問題3: 矛盾する情報源が存在

**原因**:
- 調査時期の違い
- 調査方法の違い
- サンプルの違い
- 定義の違い

**解決策**:
```
1. 各情報源の調査時期、方法、サンプルを確認
2. 使用されている定義や前提を比較
3. 最新かつ最も信頼性の高い情報源を優先
4. 矛盾がある点を明示的に記述
5. 「見解が分かれている」等、論争的性質を明記
```

#### 問題4: 信頼性スコアが低い

**原因**:
- 情報源タイプの信頼性が元々低い
- 検証項目での減点が多い
- 著者の専門性が不明

**解決策**:
```
1. 代替情報源探索機能を使用
2. より信頼性の高いソース(学術論文、公式統計等)を探す
3. 複数の中程度信頼性ソースで相互検証
4. 使用する場合は「限定的信頼性」と明記
5. 主要論拠ではなく補足的使用に限定
```

#### 問題5: 最新情報が見つからない

**原因**:
- 情報更新の遅延
- 検索方法の不適切
- 実際に最新研究が不足

**解決策**:
```
1. Google Scholarで「最近1年」等の期間指定検索
2. arXivやプレプリントサーバーを確認
3. 専門家のTwitter/ブログで最新動向確認
4. 業界ニュースサイトで最新報道確認
5. 「最新データは〇〇年時点」と明記し、更新予定を調査
```

## FAQ(よくある質問)

### Q1: すべての引用でTrustGuardを実行する必要がありますか?

**A**: 以下の優先順位で判断してください:

**必須実行**:
- 主要な論拠となる引用
- 統計データや数値情報
- 論争的な主張
- 初めて使用する情報源

**推奨実行**:
- 補足的情報
- 背景説明の引用
- 事例紹介

**省略可**:
- 周知の事実
- 教科書レベルの基礎知識
- 過去に検証済みの情報源からの追加引用

---

### Q2: 信頼性スコアが60点未満でも使用できますか?

**A**: 原則として以下の条件下でのみ使用可能:

1. **より信頼性の高い代替ソースが存在しない**
2. **補足的・参考的使用に限定**(主要論拠としない)
3. **限界を明示的に記述**
4. **可能な限り他のソースで交差検証**

記述例:
「民間ブログ(信頼性は限定的)では〇〇との記述があるが、
公式な確認はされていない」

---

### Q3: 検証に時間がかかりすぎる場合は?

**A**: 以下の効率化策を検討:

1. **クイック検証モード使用**
   - URL接続確認のみ
   - 基礎スコア算出のみ
   - 詳細検証は後で実施

2. **優先順位付け**
   - 重要度の高い引用を優先
   - 主要論拠から検証

3. **バッチ処理**
   - 複数の引用をまとめて検証
   - 同一情報源からの複数引用は一括検証

---

### Q4: 日本語の情報源はどう扱えばいいですか?

**A**: 日本語情報源も同様のプロセスで検証可能:

**対応済み機能**:
- 日本の主要データベース対応(国会図書館、CiNii等)
- 日本政府公式サイト検証
- 日本のメディア(新聞、雑誌)評価

**注意点**:
- 国際的データベース(PubMed等)では英語情報が優先
- 日本語固有の信頼性評価基準適用
- 翻訳版がある場合、原文も確認推奨

---

### Q5: 古い情報源(10年以上前)の扱いは?

**A**: 分野と使用目的により判断:

**問題なし**:
- 歴史的事実
- 古典的理論
- 基礎的科学原理
- 過去の出来事の記録

**要注意**:
- 技術的内容(急速に変化)
- 統計データ(最新版使用推奨)
- 社会的現象(時代により変化)

**推奨アクション**:
- 最新の研究で更新されていないか確認
- 「〇〇年時点では」と時点を明記
- 可能な限り最新の類似研究も併記

---

### Q6: オープンアクセスでない学術論文はどう扱う?

**A**: 以下の手順で対応:

1. **アクセス方法の確認**
   - DOI経由でのアクセス試行
   - プレプリント版の有無確認
   - 著者による自己アーカイブ確認

2. **代替手段**
   - 大学図書館経由でのアクセス
   - 著者への直接コンタクト(ResearchGate等)
   - Abstractのみ利用(本文アクセス不可の場合)

3. **記述上の対応**
   - アクセス制限がある点を明記
   - 可能な範囲での検証結果を記述
   - オープンアクセスの代替論文も併記
