# Tokyo-Cite: 東京大学スタイル引用管理システム

## Overview

Tokyo-Citeは、学術研究論文における引用と参考文献の自動生成・検証を行う統合的引用管理スキルです。東京大学の引用スタイルに厳格に準拠し、情報の透明性・検証可能性・学術的厳密性を確保します。

### Core Value Proposition

1. **自動化による効率化**: 手動引用作成（1件2-3分）を30秒に短縮
2. **品質保証**: 引用ミス・ハルシネーションの完全排除
3. **透明性確保**: すべての情報源の検証可能性を担保
4. **学術的厳密性**: 東京大学スタイルへの完全準拠

---

## Core Capabilities

### 1. 本文引用の自動変換

**入力**: 生文献情報（著者、タイトル、URL、発行日等）  
**出力**: 東京大学スタイル準拠の本文引用形式

**基本形式**: 「（著者名, YYYY.M.D, ページ）」または「（著者名, YYYY, ページ）」

**処理フロー**:
1. 著者数の自動判定（単著/共著/多数著者）
2. 日付形式の標準化（YYYY.M.D形式またはYYYY形式に統一）
3. ページ情報の処理（p./pp.の自動判定）
4. オンライン資料の識別と適切な形式選択

### 2. 参考文献リストの自動生成

**文献タイプ別フォーマット適用**:
- 書籍
- 雑誌論文
- ウェブサイト
- 政府刊行物・報告書
- 会議・学会発表

**追加機能**:
- URLの自動検証と接続確認
- アクセス日の自動記録（現在日付）
- アルファベット順/五十音順の自動ソート
- 重複文献の自動検出と除去

### 3. 引用整合性チェック

**検証項目**:
- 本文引用と参考文献リストの完全一致
- 重複引用の検出（1ソース1引用原則）
- 引用長制限の自動検証（直接引用25語以下）
- 未引用文献の警告（参考文献リストに存在するが本文に引用なし）
- 存在しない引用の警告（本文に引用があるが参考文献リストに不在）

### 4. URL永続性確保

**検証プロセス**:
1. URLの構文チェック（プロトコル、ドメイン、パス）
2. 接続テスト実行（HTTP/HTTPS）
3. ステータスコード確認（200: 正常、404/500等: エラー）

**接続不能時の自動対応**:
1. Google検索で代替URL探索
2. Internet Archive（Wayback Machine）確認
3. DOIリンク探索（学術論文）
4. 公式サイトでの代替ページ検索

**永続性強化策**:
- DOI/Permanent linkの優先使用
- Internet Archiveへの登録推奨
- 複数URLの併記オプション

## Citation Formats（本文引用形式）

### 基本パターン

#### 1. 単著者（日付詳細あり）
```
（田中, 2023.4.15, p.42）
（田中, 2023.4.15）※オンライン資料でページなし
```

#### 2. 単著者（発行年のみ）
```
（田中, 2023, p.42）
（田中, 2023）※オンライン資料でページなし
```

#### 3. 複数著者（2名）
```
（田中・山田, 2023.4.15, p.42）
（田中・山田, 2023, p.42）※発行年のみの場合
```

#### 4. 複数著者（3名以上）
```
（田中ほか, 2023.4.15, p.42）
（田中ほか, 2023, p.42）※発行年のみの場合
```

#### 5. 組織・機関
```
（経済産業省, 2023.4.15, p.57）
（経済産業省, 2023, p.57）※発行年のみの場合
（日本経済新聞, 2024.2.1）
```

#### 6. 発行日不明
```
（著者名, n.d., p.42）
※n.d. = "no date"
```

#### 7. 複数情報源の統合
```
（田中, 2023.4.15; 山田, 2022.6.10）
（田中, 2023; 山田, 2022）※発行年のみの場合
```

#### 8. 二次引用（原則避けるべき）
```
（原著者, 原著発表年, 二次引用元の著者, 二次引用元の発表年, p.42）
例: （Porter, 1985, 田中, 2023, p.25）
```

### 日付形式の使い分け

**YYYY.M.D形式（詳細日付）を使用する場合**:
- ウェブサイト記事（発行日が明記されている）
- ニュース記事
- プレスリリース
- ブログ投稿
- 会議発表（発表日）

**YYYY形式（発行年のみ）を使用する場合**:
- 書籍（発行年のみ記載）
- 学術雑誌論文（発行年のみ記載）
- 政府刊行物（発行年のみ記載）
- 年次報告書

**重要**: 情報源に記載されている日付情報に従うこと。詳細日付が不明な場合は発行年のみを使用。

### 引用ルール

**必須ルール**:
- ✅ 直接引用は25語以下に厳格制限
- ✅ 1ソース1引用原則（著作権遵守）
- ✅ パラフレーズを基本とし、直接引用は必要最小限
- ✅ 引用符「」を使用（直接引用時）
- ✅ 日付形式は情報源に記載されている形式に従う
- ❌ 長文の引用は禁止
- ❌ 同一ソースからの複数引用は禁止
- ❌ 不明な日付情報を推測して記載しない

## Reference List Formats（参考文献リスト形式）

### 1. 書籍

**形式**:
```
著者名（発行年）『書名』出版社, ページ数.
URL [アクセス日: YYYY.MM.DD]
```

**例**:
```
田中太郎（2023）『デジタル時代のブランド戦略』東京出版, pp.42-45.
https://www.example.com/digital-brand [アクセス日: 2025.11.22]
```

**注意点**:
- 発行年のみ記載（月日は不要）
- 単一ページ: p.42
- 複数ページ: pp.42-45
- 章全体: pp.42-87
- URLは必須（オンライン版がある場合）

### 2. 雑誌論文

**形式**:
```
著者名（発行年）「論文タイトル」『雑誌名』巻(号), 開始ページ-終了ページ.
URL [アクセス日: YYYY.MM.DD]
```

**例**:
```
山田花子（2022）「プライベートブランドの新展開」『マーケティングジャーナル』15(2), pp.112-125.
https://www.marketing-journal.jp/article/12345 [アクセス日: 2025.11.22]
```

**注意点**:
- 発行年のみ記載（巻号に月日情報が含まれる場合も年のみ）
- 論文タイトル: 「」で囲む
- 雑誌名: 『』で囲む
- 巻号: 15(2)形式
- DOI優先: 利用可能な場合必ず記載

### 3. ウェブサイト

**形式**:
```
著者名/組織名（発行年月日）「ページタイトル」『ウェブサイト名』
URL [アクセス日: YYYY.MM.DD]
```

**例**:
```
日本経済新聞（2024.2.1）「PBの衝撃、物価高で存在感増す」『日本経済新聞電子版』
https://www.nikkei.com/article/DGXZQOUC27DSF0X20C24A1000000/ [アクセス日: 2025.11.22]
```

**注意点**:
- 発行日: 可能な限り正確に（年月日）
- 著者不明: 組織名を使用
- 更新日: 最終更新日を記載

### 4. 政府刊行物・報告書

**形式**:
```
発行機関名（発行年）『報告書タイトル』発行機関, ページ数.
URL [アクセス日: YYYY.MM.DD]
```

**例**:
```
経済産業省（2023）『令和5年版 小売業白書』経済産業省, p.57.
https://www.meti.go.jp/report/retail2023 [アクセス日: 2025.11.22]
```

**注意点**:
- 発行年のみ記載
- 発行機関: 正式名称使用
- 年度表記: 令和X年版/20XX年版を併記可
- ページ: 参照した特定ページを記載

### 5. 会議・学会発表

**形式**:
```
発表者名（発表年月日）「発表タイトル」『会議名/学会名』開催場所.
URL [アクセス日: YYYY.MM.DD]
```

**例**:
```
佐藤一郎（2024.3.15）「小売業におけるAI活用の最新動向」『第35回流通学会年次大会』東京大学.
https://www.distribution-society.jp/conference/35th [アクセス日: 2025.11.22]
```

**注意点**:
- 発表日: 正確な日付（YYYY.M.D形式）
- 会議名: 正式名称
- 開催場所: 大学名/都市名
- プロシーディングス: URL必須（公開されている場合）

## URL Validation Protocol（URL検証プロトコル）

### 一次検証（必須）

**ステップ1: 構文チェック**
- プロトコル確認（https:// or http://）
- ドメイン妥当性確認
- パス構造確認
- 特殊文字エスケープ確認

**ステップ2: 接続テスト**
```python
# 疑似コード
import requests
response = requests.get(url, timeout=10)
if response.status_code == 200:
    return "✅ 接続成功"
else:
    return f"❌ エラー: {response.status_code}"
```

**ステップ3: コンテンツ確認**
- タイトル一致確認
- 著者名一致確認
- 発行日一致確認

### 二次検証（接続不能時）

**代替URL探索順序**:
1. **Google検索**
   - 検索クエリ: "タイトル" + "著者名"
   - 公式サイト優先
   - 最新版確認

2. **Internet Archive**
   - URL: `https://web.archive.org/web/*/[オリジナルURL]`
   - 最新スナップショット取得
   - アーカイブURLを記載

3. **DOI検索（学術論文）**
   - Crossref API使用
   - DOI形式: `https://doi.org/10.xxxx/xxxxx`
   - DOI優先記載

4. **公式サイト検索**
   - サイト内検索実行
   - 同等コンテンツ確認
   - リダイレクトURL取得

### 検証不能時の明示

**記載形式**:
```
[オリジナルURL - 2025.11.22時点で接続不能。代替URL検索実施済み]
```

**または**:
```
[接続不能 - Internet Archive保存版]: https://web.archive.org/web/...
```

### 永続性強化（推奨）

**Permanent Link優先**:
- DOI > Permanent URL > 通常URL
- 学術論文: DOI必須
- ニュース: Permanent Link使用

**複数URL併記**:
```
https://www.example.com/article [代替: https://web.archive.org/...] [アクセス日: 2025.11.22]
```

## Validation Rules（検証ルール）

### 引用長制限

**ルール**: 直接引用は25語以下

**検証方法**:
```python
def validate_quote_length(quote_text):
    word_count = len(quote_text.split())
    if word_count > 25:
        return f"❌ 引用長超過: {word_count}語（上限25語）"
    return f"✅ 引用長: {word_count}語"
```

**対処法**:
- パラフレーズに変更
- 重要部分のみ抽出
- 複数引用に分割（ただし1ソース1引用原則に注意）

### 1ソース1引用原則

**ルール**: 各情報源からは1か所のみ引用可能

**検証方法**:
```python
def check_duplicate_citations(citations):
    source_count = {}
    for citation in citations:
        source = citation.get_source_id()
        source_count[source] = source_count.get(source, 0) + 1
    
    duplicates = {k: v for k, v in source_count.items() if v > 1}
    if duplicates:
        return f"❌ 重複引用: {duplicates}"
    return "✅ 1ソース1引用原則: 遵守"
```

**対処法**:
- 複数引用を1つに統合
- パラフレーズで情報を要約
- 最も重要な箇所のみ引用

### 形式統一性

**チェック項目**:
- [ ] 著者名形式統一（姓のみ/フルネーム）
- [ ] 日付形式統一（YYYY.M.D または YYYY - 情報源に応じて）
- [ ] ページ形式統一（p./pp.）
- [ ] 括弧形式統一（全角）
- [ ] URL形式統一（http/https）
- [ ] アクセス日形式統一（YYYY.MM.DD）

### 参考文献ソート

**日本語文献**: 五十音順
- あ行 → か行 → さ行 → ...
- 同姓の場合: 名前で区別
- 同一著者: 発行年順（古→新）

**英語文献**: アルファベット順
- A → B → C → ...
- 同姓の場合: 名前で区別
- 同一著者: 発行年順（古→新）

**混在の場合**: 日本語→英語の順

## Best Practices（ベストプラクティス）

### 引用作成時

1. **早期着手**: 資料収集時に同時に引用情報を記録
2. **完全情報**: すべての必須項目を初回から記録（後で探す手間を削減）
3. **URL即時検証**: 資料収集直後にURL接続テスト実行
4. **DOI優先**: 学術論文は常にDOI確認・使用
5. **アクセス日記録**: Web資料は必ず現在日付を記録
6. **日付正確性**: 情報源に記載されている日付形式をそのまま使用（推測しない）

### 参考文献リスト作成時

1. **一元管理**: すべての引用を一箇所に集約
2. **重複チェック**: 定期的に重複文献を確認・削除
3. **自動ソート**: 完成後に五十音順/アルファベット順にソート
4. **最終検証**: 提出前にすべてのURLを再検証
5. **バックアップ**: Internet Archiveへの登録を推奨

### 整合性確保

1. **定期チェック**: 執筆中も定期的に整合性チェック実行
2. **未引用削除**: 参考文献リストから未引用文献を削除
3. **不在追加**: 本文引用に対応する参考文献を追加
4. **形式統一**: 全引用の形式を最終確認で統一
5. **査読前**: 提出前に完全な整合性チェック実施

### トラブルシューティング

**問題1: URL接続不能**
- → Internet Archive確認
- → Google検索で代替URL探索
- → 著者・出版社に直接問い合わせ
- → DOI確認（学術論文）

**問題2: 著者名不明**
- → 組織名使用
- → 編集者名使用（書籍）
- → ウェブサイト名使用（最終手段）

**問題3: 発行日不明**
- → (n.d.)使用
- → 最終更新日使用（ウェブサイト）
- → アクセス日のみ記載

**問題4: ページ番号不明**
- → オンライン資料は省略可
- → セクション番号使用（PDF）
- → パラグラフ番号使用（最終手段）

**問題5: 日付詳細不明**
- → 発行年のみ使用（YYYY形式）
- → 推測して記載しない
- → 情報源に記載されている通りに記録

## Troubleshooting（トラブルシューティング）

### よくある問題と解決策

#### 問題1: 「著者名が不明」

**症状**: ウェブサイトに著者名が記載されていない

**解決策**:
1. About/著者情報ページ確認
2. 組織名を著者として使用
3. 編集者名使用（最終手段）
4. 形式: （組織名, YYYY.M.D または YYYY）

#### 問題2: 「URLが非常に長い」

**症状**: URLが200文字以上

**解決策**:
1. Permanent Link確認
2. 短縮URL作成（bit.ly等）※非推奨
3. DOI使用（学術論文）
4. 必要に応じて改行して記載

#### 問題3: 「複数の発行日が存在」

**症状**: 初版、改訂版、オンライン版で日付が異なる

**解決策**:
1. 実際に参照した版の日付使用
2. 複数版がある場合は明記
3. 形式: （著者, 初版YYYY/改訂版YYYY）

#### 問題4: 「日本語と英語の文献が混在」

**症状**: 参考文献リストのソート方法が不明

**解決策**:
1. 日本語文献を先に記載（五十音順）
2. その後、英語文献を記載（アルファベット順）
3. セクションヘッダーで区別（オプション）

#### 問題5: 「PDF版とWeb版でページ番号が異なる」

**症状**: 同一論文のPDF版とWeb版でページ番号が不一致

**解決策**:
1. PDF版のページ番号を優先使用
2. Web版の場合は「オンライン」と明記
3. 形式: （著者, YYYY または YYYY.M.D, オンライン）

#### 問題6: 「発行年しかわからない」

**症状**: 書籍や論文で発行年のみ記載されている

**解決策**:
1. 発行年のみ使用（YYYY形式）
2. 推測して詳細日付を追加しない
3. 本文引用: （著者, YYYY, p.XX）
4. 参考文献: 著者名（YYYY）『タイトル』...
